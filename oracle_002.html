SQL 문법


DQL (Data Query Language)

데이터 조회: SELECT

데이터 질의어: 데이터베이스에서 데이터를 조회하는 명령어.

  • 데이터를 새로 생성 및 수정하지 않는 일종의 읽기 전용 작업
    (SUBQUERY문에서 SELECT 절을 사용하면 원하는 정보만 추출하여 DML 적용 가능)
  • 특정 컬럼을 선택하여 조회 가능하며, 필터링/정렬/가공 등 추가 작업 가능
    1. 필터링: WHERE
    2. 정렬: ORDER BY
    3. 가공: 산술 연산, AS "별칭", 연결 연산자 ||, 중복 제거 DISTINCT

DML (Data Manipulation Language)

데이터 조작어: 테이블 객체에 데이터(인스턴스)를 삽입, 수정, 삭제하는 명령어.

  !!! 주의 !!!

  • WHERE절 필수!
    : 서브쿼리를 쓸 때, 전체 행을 수정, 삭제할 것이 아니라면 서브쿼리 안에 WHERE절을 필수 삽입한다.
  • 데이터 타입 확인
    : 값을 삽입, 수정할 때 컬럼의 데이터 타입에 맞는 값을 입력해야 한다.
  • 데이터 확인 후 COMMIT 잊지 않기
    : DML 실행 후 SELECT절로 데이터 확인 후, 문제가 있으면 ROLLBACK, 확정 시 COMMIT 필수!

INSERT: 데이터 삽입

테이블에 새로운 행을 추가하는 명령어.

  1. 모든 컬럼에 값을 순서대로 삽입
  2. 특정 컬럼을 지정하여 삽입
    • 지정하지 않은 값은 기본값(기본값 없을 시 null) 적용
    • 기본값 설정 없음 && not null 제약 조건
      => 꼭! 컬럼을 지정하여 값을 삽입해야 함!

  서브 쿼리로 데이터 추가하기
  INSERT 뒤에 INTO를 추가하고, VALUES 키워드를 생략하면 서브쿼리에서 추출한 데이터를 행의 각 속성값으로 하여 데이터를 추가할 수 있다.
    다중 테이블에 데이터 추가
    INSERT ALL 이후 INTO 테이블명을 여러 번 쓰면 서브쿼리에서 추출한 데이터를 행의 각 속성값으로 하여 같은 데이터를 여러 테이블에 추가할 수 있다.

UPDATE: 데이터 수정

테이블에 저장되어 있는 기존 데이터를 수정하는 명령어.

WHERE절로 수정할 대상 행을 선정하고, SET 뒤에 수정할 대상 컬럼을 지정할 수 있다.
컬럼에 수정할 값을 직접 입력할 수도 있고, 계산식을 사용하여 원래 값의 몇 배나 원래 값에서 특정값을 더하거나 뺀 값으로도 설정할 수 있다.

DELETE: 데이터 삭제

테이블의 기존 행을 삭제하는 명령어.

WHERE절로 삭제할 대상 행을 선정하여 삭제할 수 있다.

  !!! 주의 !!!

  • 외래키(FOREIGN KEY) 제약조건으로 묶인 부모 행 삭제 시 자식 테이블 행을 삭제하거나, 자식 테이블 행의 값을 모두 null값으로 비워야 한다.
    • 자식 테이블에서 외래키 제약조건을 ON DELETE SET NULL로 설정했다면 부모 테이블의 행 삭제 시 자식 테이블의 행이 자동으로 null값으로 비워진다.
    • 자식 테이블에서 외래키 제약조건을 ON DELETE CASCADE로 설정했다면 부모 테이블의 행 삭제 시 자식 테이블의 행이 자동으로 삭제된다.

DDL (Data Definition Language)

데이터 정의어: 데이터베이스에서 테이블, 뷰, 인덱스 등 객체의 구조를 정의/변경/삭제할 때 사용하는 SQL 명령어.

CREATE (객체 정의 및 생성)

CREATE 뒤에 생성할 객체의 타입(TABLE, VIEW, SEQUENCE 등)을 적고 객체명을 적는다.

  1. TABLE 생성
    • CREATE TABLE 문법
      • CREATE TABLE TN_NAME (
        COL_NAME DATA_TYPE[ DEFAULT DEFAULT-VALUE CONSTRAINT]
        ...
        [TABLE_LEVEL_CONSTRAINT]
      );
    • 제약조건
      • NOT NULL: NULL값 입력 제한
      • UNIQUE: 중복값 입력 제한
      • CHECK: 입력값 범위 제한
      • PRIMARY KEY: 테이블 대표 식별자(NOT NULL + UNIQUE)
      • FOREIGN KEY: 참조 데이터 간 관계 정의를 통한 참조 무결성 보장
        • 주의!
        참조하려는 부모 테이블 컬럼은 반드시 PK 또는 UNIQUE
    • DEFAULT값 설정
      • 데이터 삽입 시 별도의 값 입력 없이도 자동으로 채워질 기본값을 지정하는 기능.
  2. VIEW 생성
    • CREATE VIEW 문법
      • CREATE[ OR REPLACE][ FORCE | NOFORCE] VIEW VW_NAME [(COL_NAME, COL_NAME, ...)]
      AS SUBQUERY
      [WITH CHECK OPTION]
      [WITH READ ONLY]
  3. SEQUENCE 생성
    • CREATE SEQUENCE 문법
      • CREATE SEQUENCE SEQ_NAME
      [START WITH FIRST NUM] : 기본값 초항 = 1
      [INCREMENT BY COMMON_DIFFERENCE] : 기본값 공차 = 1
      [MAXVALUE maxval]
      [MINVALUE minval]
      [CYCLE] : 기본값 NOCYCLE
      [CACHE BYTE_NUM | NOCACHE] : 기본값 캐시 20
  4. TRIGGER 생성
    • CREATE TRIGGER 문법
      • CREATE TRIGGER TRG_NAME
      BEFORE | AFTER
      INSERT | UPDATE | DELETE ON TB_NAME
      [FOR EACH ROW]
      [WHEN 조건식]
      DECLARE
      BEGIN
      EXCEPTION
      END;
      /

ALTER (객체 변경)

ALTER 뒤에 수정할 객체의 타입을 적고 객체명을 적는다.

  1. TABLE 수정
    • ADD: 새로운 컬럼, 제약조건 추가
    • MODIFY: 컬럼의 데이터 타입, 크기, 또는 DEFAULT값을 변경할 때 사용한다.
    • DROP: 특정 컬럼을 물리적으로 완전히 삭제하거나 제약조건을 제거한다.
    • RENAME: 테이블이나 컬럼명이 잘못 지어졌거나 개편이 필요할 때 명칭을 변경합니다.
  2. SEQUENCE 수정
    3. ALTER SEQUENCE SEQ_NAME
    변경할_옵션


    !!! 주의 !!! 초항은 변경 불가!

DROP (객체 삭제)

DROP 뒤에 삭제할 객체의 타입을 적고 객체명을 적는다.
DROP 객체타입 객체명;으로 데이터베이스에서 객체 구조와 그 안에 담긴 모든 데이터를 영구적으로 삭제한다.

!!! 주의 !!!

DDL 실행 후에는 자동으로 DML이 COMMIT되므로 ROLLBACK이 필요한 사항은 없는지 확인 필수!!!

선택 과제

테이블을 왜 분리해야 하는가?

상황: 하나의 ‘주문 테이블’에 [고객 이름, 연락처, 주문 상품, 가격, 배송지]를 모두 몰아서 저장하고 있습니다.

**생각해볼 부분**
  1. 한 고객이 이사를 가거나 전화번호를 바꾸면 어떤 번거로운 일이 생길까?
    2. 상품 주문 내역을 관리할 때 배송지와 전화번호 변경 전의 주문 내역은 변경 전 정보대로 저장해야 하는데, 한 테이블에서 고객정보와 주문 내역을 전부 관리하게 되면 변경 직후 시점 주문 상품부터 전부 일일이 수정해야 하는 번거로움이 있다.
    (주문 내역이 아닌 단순 고객 정보로서 저장한다고 해도 성능 면에서 문제가 생길 수 있다. WHERE절로 고객 ID 등을 조건으로 넣어 일괄수정할 수는 있지만, 대량 주문하는 고객일수록 데이터가 많기 때문이다.)
    +) 일반적으로 한 고객은 여러 번 주문을 하는데, 똑같은 고객 이름, 연락처, 배송지가 여러 행에 중복되어 저장되는 것도 메모리 공간 낭비이다.
  2. 아직 상품을 한 번도 주문하지 않은 신규 가입 회원의 정보는 이 테이블에 저장할 수 있을까?
    3. 주문 상품 컬럼과 가격 컬럼에 NOT NULL 제약조건이 있는 경우 저장할 수 없다. 주문 상품 컬럼, 가격 컬럼의 기본값을 0으로 설정했다면 저장할 수 있다.
    하지만 주문 테이블은 '주문 내역'을 관리하는 것이 목적이므로, 이와 같은 예비 데이터를 저장하는 것은 저장공간을 낭비하는 셈이 되어 비효율적이다.
    +) 주문 내역을 저장하는 목적의 테이블이므로 주문 상품 컬럼과 가격 컬럼이 NULL 허용이라면 테이블 설계를 잘못한 것으로 보인다. 관리하려는 대상의 중요한 속성이므로 NULL 비허용일 가능성이 높고, 이럴 경우 신규 회원 정보는 저장되지 않으므로 데이터 관리 면에서도 오점이다.
  3. 이 문제를 해결하려면 테이블을 어떻게 나누는 것이 좋을까?
    4. 고객 정보를 저장하는 테이블과 상품 정보를 저장하는 테이블, 주문 내역을 관리하는 테이블을 따로 생성하여 관리한다. 이때 주문 내역 테이블에서는 주문자 컬럼(고객 테이블의 PK인 고객 ID를 참조), 주문상품 컬럼(상품 테이블의 PK인 품번을 참조), 주문번호 컬럼(주문 내역 PK)을 포함하도록 한다.
